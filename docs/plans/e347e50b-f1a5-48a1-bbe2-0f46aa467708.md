## Goal
Build the "CyOpsArenaCookbook" web app: a single React + Express + SQLite product that gives CyOpsxMinimax hackathon participants a rules/prize/scoring guide, a Brainstorming-skill-driven AI prompt generator, a live vote ticker, zh/en i18n, and a global AI-credit pool that grants +100 per vote and deducts 20 per AI request.

## Assumptions
- The token-AI endpoint is OpenAI-compatible; the executor will probe `GET /v1/models` at boot, prefer `gpt-4o-mini`, then `gpt-4o`, then the first model id returned, and cache the chosen id in `apps/api/src/services/llm.selectedModel.txt`.
- The credit pool is a single global ledger row that starts at 1000 and is mutated only by `+100` per observed net-new vote and `-20` per completed AI request; no periodic reset, no per-user accounts, no admin top-up.
- The submissions API response is JSON and exposes a numeric vote count; the normalizer will accept `votes`, `vote_count`, `total_votes`, or `data.votes` and skip a poll cycle (with a warning log) when none of them are present.
- "Each AI request" means each successful assistant turn, not each user keystroke; failed/aborted calls do not deduct credits.
- The official brainstorming skill is fetched once at build time via `agent-browser` and saved to `apps/api/src/services/brainstorming-skill.md`; an embedded condensed copy lives at `apps/api/src/services/brainstorming-skill.embedded.md` for offline use.
- The local dev story is `pnpm dev` (Vite on `:5173` with a `/api` proxy to Express on `:4000`); deployment is optional and addressed in the Deployment section.
- The session is read-only for the two reference markdown files: the Guide page renders content sourced from them, the prompt generator injects the rubric weights, and copies are placed under `docs/references/` as evidence.
- The user-facing app is one global instance shared by all participants; no login, no per-user credit balances.

## Acceptance Criteria
- AC-1: Project scaffold and config.
  - pnpm workspaces (`pnpm-workspace.yaml`) with `apps/web` (Vite + React 18 + TS + Tailwind) and `apps/api` (Node 20 + Express + TS + `better-sqlite3` + zod + vitest).
  - Root scripts: `dev`, `build`, `lint`, `test`, `seed`, `migrate`.
  - `.env.example` declares `PORT`, `DATABASE_FILE`, `CORS_ORIGIN`, `OPENAI_BASE_URL=https://token-ai.cysic.xyz/v1`, `OPENAI_API_KEY=`, `OPENAI_MODEL=`, `SUBMISSIONS_URL=https://arena.cysic.xyz/api/submissions/7YCp4WI_czoa-4MnZ-aBC`, `VOTE_POLL_MS=60000`, `CREDIT_START=1000`, `CREDIT_PER_VOTE=100`, `CREDIT_PER_CHAT=20`.
  - `.env` is created from `.env.example` with the real API key; `git check-ignore -v .env` reports the ignore rule and `git ls-files .env` is empty.
  - A `secret-guard.sh` pre-commit hook blocks any commit that adds `.env`.
- AC-2: SQLite schema, migrations, seed.
  - Tables: `credit_ledger(id, balance, delta, reason, ref, created_at)`, `chat_log(id, session_id, role, content, tokens_in, tokens_out, created_at)`, `vote_snapshot(id, votes, raw_json, observed_at)`, `prompt_draft(id, track, answers_json, prompt, created_at)`.
  - `apps/api/src/db/migrations/0001_init.sql` runs on boot via `db/migrate.ts`; `db/seed.ts` inserts the starting 1000-credit row.
- AC-3: Vote polling and ticker endpoint.
  - `apps/api/src/jobs/votePoller.ts` runs every `VOTE_POLL_MS`, upserts `vote_snapshot`, and on positive deltas writes a `credit_ledger` row with `delta=+100*(new-old)` and broadcasts via SSE.
  - `GET /api/votes` returns `{ current, history, lastDelta, observedAt }`.
  - `GET /api/votes/stream` is an SSE endpoint emitting `{ type:'vote', delta, current, observedAt }` and `{ type:'credits', balance, blocked }` events.
  - Polling never blocks the server: failures use exponential backoff (1m → 5m cap) and a single in-flight flag.
- AC-4: Global credit pool with concurrency-safe enforcement.
  - `GET /api/credits` returns `{ balance, perVote:100, perChat:20, blocked: balance<20 }`.
  - `POST /api/chat` and `POST /api/prompt/*` call a single SQL `UPDATE credit_ledger SET balance = balance - 20 WHERE id = (SELECT id FROM credit_ledger ORDER BY id DESC LIMIT 1) AND balance >= 20` inside a transaction; on `changes()===0` the response is `402 Payment Required` with `{ error:'insufficient_credits', balance }`.
  - SSE broadcasts `{ type:'credits', balance, blocked:true }` whenever `balance` first drops below 20 or first returns to ≥ 20.
  - A concurrent-deduction test (5 parallel POSTs from a 100-credit state) ends with `balance=0` and exactly five 402/200 responses in some order; balance never goes negative.
- AC-5: LLM service against token-ai.cysic.xyz.
  - `apps/api/src/services/llm.ts` exports `chat(messages, opts)`; on boot it calls `GET /v1/models`, picks the first id of `gpt-4o-mini`/`gpt-4o`/first available, caches it, and exposes `/v1/chat/completions` calls.
  - All calls enforce an 8k/2k token cap and a 30 s timeout; failures throw `LlmError`; the route translates it to a sanitized `502` JSON body and never leaks the API key.
  - `chat_log` rows are written with `tokens_in`, `tokens_out`, and `latency_ms` for every successful call.
- AC-6: Brainstorming-skill-driven prompt generator.
  - The full upstream skill text is cached at `apps/api/src/services/brainstorming-skill.md` (built by `apps/api/scripts/fetch-brainstorming-skill.sh` using `agent-browser open https://github.com/obra/superpowers/blob/main/skills/brainstorming/SKILL.md && agent-browser snapshot > ...`); an embedded copy at `brainstorming-skill.embedded.md` is the offline fallback.
  - State machine: `INTENT → CONTEXT → CONSTRAINTS → APPROACHES → DESIGN → REFINE → DONE`. Each transition is one LLM call with a system prompt that includes the cached skill text, the running transcript, and an explicit "ask exactly one question" instruction; the LLM is asked to respond in the user's chosen locale.
  - `POST /api/prompt/start` accepts `{ track?: 'ship-a-feature'|'mcp-server'|'whole-repo-refactor'|'resurrection'|null, freeText?: string, locale?: 'en'|'zh' }` and returns `{ sessionId, question, step:0 }`; the first question is the Socratic "what is the user actually trying to do?" question from the skill.
  - `POST /api/prompt/answer` accepts `{ sessionId, answer }`; on the `DONE` step it returns `{ done:true, prompt, sections: { problem, users, solution, approach, tradeoffs, scorecard } }` and persists the `prompt_draft` row.
  - The emitted `prompt` is a CyOps-ready block whose front-matter is validated by a zod schema and includes: `track`, `target_audience`, `success_criteria[]`, and `rubric_checklist` with the six dimensions and weights from `AI-scoring-breakdown.md` (`implementation_engineering_quality:20, architecture_complexity_fit:16, deliverable_completeness:20, project_copy_documentation:16, ai_agent_integration:20, implementation_innovation:8`).
  - `POST /api/prompt/start` and each `/answer` deduct 20 credits only on successful AI calls; on `402` the server keeps the session state and returns the current question.
- AC-7: Frontend pages and routing.
  - Routes (react-router): `/` (Guide), `/prompt` (Prompt Studio), `/vote` (Live Ticker), `/about` (About).

## Implementation Notes
- Stack: pnpm workspaces; Vite 5 + React 18 + TS + Tailwind + react-router + zustand + i18next + react-i18next; Express 4 + TS + better-sqlite3 + zod + vitest; SSE via chunked `res.write`.
- Backend layout: `apps/api/src/{db,services,routes,jobs,middleware,schemas}/` with single-responsibility modules: `llm.ts`, `credits.ts`, `votes.ts`, `brainstorming.ts`.
- Credit ledger invariants: every mutation goes through one SQL `UPDATE ... WHERE balance >= ?` inside `BEGIN ... COMMIT`; the application never reads-then-writes balance without the guard.
- LLM model selection: probe `GET /v1/models` at boot, prefer `gpt-4o-mini`, then `gpt-4o`, then the first id; cache into `apps/api/src/services/llm.selectedModel.txt`; allow override via `OPENAI_MODEL`.
- Brainstorming skill ingestion: `apps/api/scripts/fetch-brainstorming-skill.sh` uses `agent-browser open https://github.com/obra/superpowers/blob/main/skills/brainstorming/SKILL.md && agent-browser snapshot` and writes the markdown to `apps/api/src/services/brainstorming-skill.md`; an embedded copy at `brainstorming-skill.embedded.md` is used when the script fails.
- Brainstorming state machine: each transition is a single LLM call with a system prompt that includes the cached skill text, the running transcript, the chosen locale, the track name (if any), and the rubric weights. The `DONE` transition is forced to emit JSON that matches a zod schema before being wrapped into the CyOps prompt format.
- Final `prompt` format (validated by zod):
  ```
  ---
  track: <one of four>
  target_audience: <string>
  success_criteria:
    - <bullet>
  rubric_checklist:
    implementation_engineering_quality: 20
    architecture_complexity_fit: 16
    deliverable_completeness: 20
    project_copy_documentation: 16
    ai_agent_integration: 20
    implementation_innovation: 8
  ---
  # <project title>
  ## Problem
  ## Users
  ## Solution
  ## Approach
  ## Tradeoffs
  ## Self-score against rubric
  ```
- Frontend prompt studio uses a stepper with back/forward; the client tracks index locally; `POST /api/prompt/answer` is append-only and idempotent on duplicate `sessionId`.
- Live ticker uses one `EventSource('/api/votes/stream')` per page; the right-side toast animates with a Tailwind `translate-x`/`opacity` keyframe and auto-dismisses at 6 s.
- Vote normalizer: try `votes` → `vote_count` → `total_votes` → `data.votes`; if none parse, log a warning and skip the ledger write for that cycle (no credit grant, no SSE event).
- Secrets: `OPENAI_API_KEY` lives only in `.env` (gitignored) and `.env.example` is committed with an empty value; pre-commit `secret-guard.sh` rejects any commit that adds a `.env`-tracked file.
- Web research with `agent-browser`:
  - `agent-browser open https://arena.cysic.xyz/` to capture the live UI (HTML + screenshot) into `docs/references/arena/`.
  - `agent-browser open https://github.com/obra/superpowers/blob/main/skills/brainstorming/SKILL.md` + `agent-browser snapshot` to cache the skill into `apps/api/src/services/brainstorming-skill.md`.
  - The two reference markdown files are copied verbatim into `docs/references/` as `AI-scoring-breakdown.md` and `RecommendedTrackCategories-Scoring-rubric.md`.
- Build order (TDD):
  1. Scaffold workspaces, `.env*`, `.gitignore`, husky guard, CI script.
  2. SQLite migration + seed (red→green test for `migrate.ts`).
  3. Credits service + ledger tests (including the concurrent deduction invariant).
  4. LLM service with model discovery + error mapping.
  5. Vote poller + `/api/votes` + SSE.
  6. Brainstorming state machine + cached skill loader + `/api/prompt/*`.
  7. Frontend layout, i18n, Guide, Prompt Studio stepper, Vote Ticker with SSE.
  8. README + `docs/references/` + `agent-browser` captures.
  9. End-to-end smoke via `agent-browser`.
- Risk controls:
  - Token-AI rate limits: 8k/2k token cap, 30 s timeout, single in-flight request per session id.
  - SSE backpressure: server flushes every event; client reconnects with 5 s jittered backoff.
  - Vote API drift: normalizer + skip-and-log, no crash.
  - Concurrency: single SQL guard for credit mutations, no application locks.
  - Brainstorming upstream outage: cached file + embedded copy, both versioned in the repo.
  - API key exposure: response body sanitized by a shared `safeLlmError()` helper; logs redact the key.

## Out of Scope
- User accounts, authentication, per-user credit balances (single global pool only).
- Persistent multi-session chat history UI (the `chat_log` table exists for audit, not for re-opening sessions).
- Direct posting of submissions to the CyOpsxMinimax arena API.
- Production HTTPS termination, Docker, k8s, monitoring (only a basic `pnpm dev`/test setup).
- Real-time admin dashboards or top-up tools for the credit pool.
- Localizations beyond zh and en.
- Editing the official scoring rubric at runtime (loaded from the markdown file at build time).
- Voting/anti-abuse controls on the cookbook side (we trust the upstream arena).

## Deployment Suggestions & Platform Detection
- The product is a standard web app: Vite + React frontend and an Express + SQLite backend. Vercel is suitable for the **frontend** (`apps/web`) — once the repo is connected, the "Deploy" tab in the right-hand panel will run `pnpm --filter web build` and serve the static `dist/` output, with the `/api` calls configured via `VITE_API_BASE_URL` to point at the separately hosted backend.
- The **backend** (Express + `better-sqlite3`) is not a fit for Vercel's serverless functions because of the long-lived `setInterval` vote poller, SSE connections, and a local SQLite file. Recommended hosts: any small VM or container (Render, Fly.io, Railway, a $5 VPS) with `pnpm --filter api start` as the start command and a persistent disk mount for `apps/api/data/cookbook.sqlite`.
- Cloudflare Pages is also viable for the frontend; Cloudflare Workers is not a fit for the backend for the same SSE/long-polling reasons.
- For local hackathon demo, just `pnpm dev` is enough; the README's "Quick start" section should call out the Vercel one-click deploy button for the frontend so judges can spin up a public preview quickly.
